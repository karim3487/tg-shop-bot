# Тестовое задание: Telegram Shop Bot

## Описание
Нужно сделать интернет-магазин, состоящий из трех сервисов:
1. **Telegram-бот** (Aiogram 3)
2. **Админ-панель** (Django)
3. **WebApp** (React/TypeScript или Next.js) — встраивается в Telegram

Как сервисы будут общаться между собой (общая БД, REST API, shared-пакет и т.д.) — решаете сами. На собеседовании попросим объяснить, почему выбрали именно такой подход. Каждый сервис должен запускаться независимо и иметь свой набор зависимостей.

## Стек:

| Сервис | Технологии |
| :--- | :--- |
| **Telegram Bot** | Python 3.12+, Aiogram 3.x |
| **Admin Panel** | Python 3.12+, Django 5.0+ |
| **WebApp** | React (TypeScript) или Next.js |
| **БД** | PostgreSQL |
| **Деплой** | Docker, docker-compose |

## Что нужно сделать

### 1. Telegram Bot

* **1.1. Регистрация:**
    * По команде `/start` бот просит поделиться контактом через `KeyboardButton(request_contact=True)`.
    * Телефон сохраняется в профиль.
    * При повторном `/start` контакт не запрашивается.
* **1.2. Проверка подписки (Middleware):**
    * Написать middleware, который на каждый апдейт проверяет подписку пользователя на нужный канал/группу.
    * Без подписки бот не дает доступ, предлагает подписаться.
    * Список каналов управляется через Django-админку (не через `.env`).
* **1.3. Каталог:**
    * Категории и подкатегории на inline-кнопках.
    * Пагинация (кнопки “Назад” / “Вперед”).
    * Карточка товара: название, описание, цена, кнопка “В корзину”.
    * У товара может быть несколько фото — отправлять через `MediaGroup`.
    * Кнопка для открытия каталога в WebApp.
    * Callback data через CallbackData Factory.
* **1.4. Deep Linking:**
    * Можно поделиться ссылкой на товар: `t.me/botname?start=product_<id>`.
    * По такой ссылке бот сразу показывает карточку товара.
* **1.5. Корзина:**
    * Список товаров с итоговой суммой.
    * Изменение количества (+/-).
    * Удаление позиций.
    * Очистка корзины.
    * Кнопка “Оформить заказ”.
* **1.6. Оформление заказа (FSM):**
    * Сделать через Finite State Machine (aiogram FSM).
    * Шаги: ФИО -> адрес доставки -> подтверждение.
    * Телефон подставляется из профиля автоматически (см. п. 1.1).
    * После подтверждения — создание платежа (заглушка допустима).
    * Кнопка “Я оплатил(а)” -> смена статуса заказа.
* **1.7. Уведомления:** Когда админ меняет статус заказа (из админки или из админ-чата), бот шлет пользователю сообщение с новым статусом.
* **1.8. Админ-чат:**
    * Новый заказ автоматически улетает в Telegram-группу для админов с деталями.
    * Админы могут менять статус заказа прямо из чата через inline-кнопки.
    * Команда для просмотра активных заказов.
    * ID чата настраивается в Django-админке.
* **1.9. FAQ через Inline Query:**
    * Поиск по FAQ через `inline_query` (`@botname запрос` в любом чате).
    * Результаты в формате `InlineQueryResultArticle`.
    * Пустой запрос показывает популярные вопросы.
    * По клику отправляется ответ.
* **1.10. Рассылки:**
    * Бот отправляет рассылки, которые создаются в админке.
    * Текст + картинка.
    * Отправка в фоне.
* **1.11. Команды бота:** При старте бота регистрировать команды через `set_my_commands`. Минимум: `/start`, `/catalog`, `/cart`, `/help`.
* **1.12. Middleware (обязательные):** Минимум 2 кастомных middleware:
    * *Проверка подписки* на канал (п. 1.2).
    * *Регистрация пользователя* — на каждый апдейт проверяет/создает пользователя в БД и прокидывает объект клиента в хендлеры через `data`.
    * Логирование апдейтов (telegram_id, тип, timestamp) тоже должно быть в проекте, но реализация на ваше усмотрение (middleware, декоратор, логгер).
* **1.13. Custom Filters:** Минимум один кастомный фильтр (например, `IsAdmin` — проверка, что пользователь админ).

### 2. WebApp (React/TypeScript или Next.js)

* **2.1. Каталог:**
    * Категории и товары с картинками.
    * Фильтр по категориям.
    * Поиск по названию.
* **2.2. Корзина:**
    * Добавление/удаление товаров.
    * Изменение количества.
    * Итоговая сумма.
    * Корзина общая с ботом (синхронизация).
* **2.3. Оформление заказа:**
    * Форма для данных доставки.
    * `Telegram.WebApp.MainButton` для подтверждения.
    * Отправка данных на бэкенд.
* **2.4. Интеграция с Telegram:**
    * Валидация `initData` на бэкенде (проверка подписи).
    * Стилизация через `themeParams`.
    * Работа с `BackButton`, `MainButton`.

### 3. Django Admin Panel

* **3.1. Каталог:**
    * CRUD категорий (с вложенностью).
    * CRUD товаров, к каждому можно прикрепить несколько фото (модель `ProductImage` или inline-формы).
    * Фильтрация и поиск.
* **3.2. Клиенты:**
    * Список клиентов (telegram_id, username, телефон).
    * История заказов клиента.
    * Статистика: сколько заказов, на какую сумму.
* **3.3. Заказы:**
    * Список с фильтром по статусу.
    * Смена статуса (с уведомлением пользователя, п. 1.7).
    * Детальный просмотр позиций.
    * Экспорт оплаченных заказов в Excel (кнопка в админке).
* **3.4. FAQ:**
    * CRUD вопросов и ответов.
    * Включение/выключение отдельных вопросов.
* **3.5. Рассылки:**
    * Создание рассылки (текст + картинка).
    * Статусы: черновик -> готово -> отправлено.
    * Результат: сколько доставлено, сколько ошибок.
* **3.6. Настройки бота:**
    * Каналы/группы для обязательной подписки.
    * ID админ-чата.
    * Настройки должны применяться без перезапуска бота.

## Технические требования

* **Архитектура:** Бот, админка и WebApp — три отдельных сервиса со своими зависимостями. Как они взаимодействуют — на ваше усмотрение. В `README` опишите и обоснуйте выбранную архитектуру.
* **Асинхронность:** Все обращения к БД из бота должны быть асинхронными. Блокировка event loop синхронными вызовами не допускается.
* **Инфраструктура:**
    * `docker-compose.yml` — весь проект поднимается одной командой.
    * `.env` для секретов (токены, пароли). Бизнес-настройки (каналы, чат) — в админке.
    * Приложить `.env.example`.
    * Логирование в файлы с ротацией.
    * Миграции БД.
